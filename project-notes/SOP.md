# 新项目快速学习 SOP（Agent 执行版）

> **用途**：本 SOP 定义了 Agent 辅助研发快速了解一个陌生 GitHub 项目的标准流程和产出规范。
>
> **目标用户**：全栈研发同学
>
> **产出物**：学习笔记（必选）+ 最简 Demo（可选，成本低才做）

---

## 0. Agent 工作原则

### 0.1 面向研发的内容偏好

**❌ 不要写这些（噪音）：**
- "支持 1000+ 集成节点" 这类纯数字堆砌、没有具体含义的描述
- 营销话术、官方 Slogan、奖项排名
- 过于细枝末节的功能列表（比如列出所有 50 个触发器节点）
- 对研发没有信息量的套话（"强大的"、"灵活的"、"企业级的"）

**✅ 要重点写这些（研发视角有价值）：**
- 它解决了什么**技术问题**？（不是业务问题，是技术上的痛点）
- 核心**架构设计**是什么？（有什么值得借鉴的）
- 数据怎么**流转**的？（输入 → 处理 → 输出的完整链路）
- **扩展性**如何？（是拿来直接用，还是要二开？）
- 对 **AI Agent 开发、工作流编排、上下文管理** 有什么借鉴价值？

### 0.2 产出物决策树

```
拿到一个 GitHub 项目
    │
    ▼
先产出【学习笔记】（必选）
    │
    ├─► 研发看完判断："这个项目对我有用吗？"
    │
    └─► 关于 Demo：
        ├─► Demo 成本低（比如纯前端、单文件）→ 直接做
        ├─► Demo 成本中等 → 在笔记里给出伪代码 + 项目结构
        └─► Demo 成本高（可视化编辑器、复杂引擎）→ 不做，等研发说感兴趣再做
```

### 0.3 学习笔记的写作原则

1. **先结论后细节**：开头先给一句话总结，让研发 10 秒内知道这是啥
2. **研发视角**：从开发者角度看价值，不是从产品经理角度
3. **抓主干放细节**：核心原理讲清楚，边角功能一笔带过
4. **诚实判断价值**：如果只是"扩充知识面"级别的，直接说实用价值不大

## 1. 产出学习笔记的标准流程

### 步骤 1：项目速览（信息收集）

**Agent 要做的事：**

1. 打开 GitHub 项目主页，读取 README
2. 搜索项目的官方文档、架构文档
3. 快速浏览目录结构，判断项目类型
4. 搜索技术博客 / 架构解读文章（如果有）

**收集的关键信息：**
- 项目一句话介绍
- 技术栈（前端/后端/数据库/核心依赖）
- Star 数、最近更新时间（判断活跃度）
- 核心架构图（如果有）
- 启动方式（最简的那种）

## 2. 多项目筛选流程

当研发给出一个关键词或方向，搜出来多个项目时：

### 步骤 1：快速筛选

Agent 先做一轮快速筛选，选出 **3-5 个最相关的项目**。

筛选标准：
- Star 数（侧面反映质量）
- 最近更新时间（是否还在维护）
- 与需求的匹配度
- 技术栈匹配度（研发熟悉的技术栈优先）

### 步骤 2：产出极简速览

对每个项目，产出一段 **100 字以内的速览**，包含：
- 一句话是什么
- 核心亮点 / 独特之处
- 对研发的大概价值

格式：
```
## 项目 1：xxx
- **一句话**：...
- **亮点**：...
- **价值预估**：高/中/低 - 原因...
```

### 步骤 3：等研发选定

研发选了某个项目后，再按照【步骤 2】产出完整的学习笔记。

## 3. 输出风格要求

### 3.1 语气
- 客观、直接，不用客套话
- 像技术同事分享心得，不是官方文档翻译
- 可以有主观判断（但要有依据）

### 3.2 长度
- 完整学习笔记：1500-3000 字
- 项目速览：每个项目 100 字以内
- 重点突出，不要凑字数

### 3.3 技术深度
- 核心原理要讲透
- 边角功能一笔带过
- 有架构图和数据流图（用 ASCII 画就行）

## 4. 常见问题处理

### Q：项目太复杂，搞不懂核心原理怎么办？
A：如实说明"核心原理较复杂，建议深入源码学习"，然后把能搞懂的外围部分讲清楚，给出深入学习的路径建议。

### Q：官方文档写得很烂，信息不够怎么办？
A：搜技术博客、GitHub Issue、源码注释。如果实在信息少，如实说明"公开资料有限，建议直接读源码"。

### Q：Demo 做不做怎么判断？
A：问自己两个问题：
1. 写一个能跑的最简版，能不能在 1 小时内搞定？
2. 这个 Demo 对理解核心原理帮助大不大？

两个都是"是" → 做
有一个"否" → 给伪代码和结构就行
两个都是"否" → 不做
