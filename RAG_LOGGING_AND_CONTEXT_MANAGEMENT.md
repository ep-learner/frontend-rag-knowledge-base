# RAG 日志系统与上下文管理学习笔记

本文档记录 RAG 系统的日志实现、Token 消耗优化、多轮对话上下文管理等关键技术点。

## 一、日志系统设计

### 1.1 为什么需要日志系统？

**核心问题**：
- 如何排查模型幻觉？（是模型问题还是知识库检索问题？）
- 如何分析 Token 消耗？（每次问答消耗多少 Token？）
- 如何回放历史对话？（了解完整的多轮对话上下文）

**解决方案**：
- 实现完整的问答日志记录系统
- 记录每次问答的完整上下文
- 提供日志查看和分析功能

---

### 1.2 日志记录内容

每次问答记录以下信息：

```json
{
```

**关键字段说明**：

| `messages` | 发送给 API 的完整上下文 | 了解多轮对话历史 |
| `tokenUsage` | Token 消耗统计 | 成本分析、优化依据 |
| `processingTime` | 处理时间 | 性能分析 |

### 1.3 日志存储方案

**存储位置**：`server/logs/chat-YYYY-MM-DD.json`

**存储策略**：
- 按日期分文件存储
- 保留 30 天的日志
- 启动时自动清理旧日志

**文件格式**：JSON 数组，每条日志是一个对象

**优点**：
- ✅ 易于查看和分析
- ✅ 支持按日期检索
- ✅ 自动清理，避免无限增长

## 二、Token 消耗分析

### 2.1 Token 消耗来源

**每次问答的 Token 消耗**：

| 来源 | Token 数量 | 说明 |
|------|-----------|------|
| Embedding API | ~250 tokens | 生成查询向量（语义检索） |
| System Context | ~1000 tokens | 知识库检索结果（固定） |
| History Context | ~500 tokens/轮 | 多轮对话历史（线性增长） |
| User Query | ~50-100 tokens | 用户问题 |
| LLM Response | ~500-1000 tokens | LLM 生成的回答 |

**总消耗**：
- 第1轮：约 1800-2300 tokens
- 第2轮：约 2300-2800 tokens
- 第10轮：约 5500-6000 tokens

---

### 2.2 Token 消耗增长趋势

**问题**：多轮对话的 Token 消耗线性增长

```
第1轮：Input = 1000 (知识库) + 100 (问题) = 1100 tokens
第2轮：Input = 1000 + 500 (第1轮对话) + 100 = 1600 tokens
第3轮：Input = 1000 + 1000 (第1-2轮对话) + 100 = 2100 tokens
第10轮：Input = 1000 + 4500 (9轮对话) + 100 = 5600 tokens
```

**成本增长**：
- 第1轮：约 ¥0.02
- 第10轮：约 ¥0.06（增长 3倍）

---

### 2.3 Token 消耗优化方案

**方案对比**：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **滑动窗口**（当前） | 简单、成本低 | 可能丢失重要信息 | 短对话 |
| **摘要压缩** | 保留关键信息、成本低 | 需要额外 API 调用 | 长对话 |
| **语义检索历史** | 智能筛选、效果好 | 复杂、需要 embedding | RAG 系统 |
| **重要性筛选** | 动态调整、灵活 | 需要评分算法 | 复杂对话 |

**推荐方案**：语义检索历史 + 滑动窗口

## 三、多轮对话上下文管理

### 3.1 当前实现方式

**Minimax API 是无状态的**：
- API 本身不保存对话历史
- 后端通过 `messages` 数组手动构建完整上下文

**实现流程**：

```
前端维护对话历史 → 每次请求携带 → 后端截取最近 N 轮 → 传入大模型
```

**关键代码**（minimax.ts）：

```typescript
async generate(prompt, context, options, history) {
  const messages: Message[] = [];

  // 1️⃣ System 消息（包含知识库内容）
  if (context) {
    messages.push({
      role: 'system',
      content: `基于以下知识库内容回答问题：\n\n${context}\n\n...`,
    });
  }

  // 2️⃣ 历史消息（多轮对话的关键！）
  if (history && history.length > 0) {
    messages.push(...history);  // ← 历史对话在这里传递
  }

  // 3️⃣ 当前用户消息
  messages.push({
    role: 'user',
    content: prompt,
  });

  // 4️⃣ 发送给 Minimax API
  return this.callApi(messages, options);  // ← 完整的上下文
}
```

---

### 3.2 为什么不在后端存储历史？

**设计权衡**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **前端存储**（当前） | 无状态设计、便于水平扩展、前端控制灵活 | 前端需要维护状态 |
| **后端存储** | 前端简单、统一管理 | 会话管理复杂、后端内存占用、难以扩展 |

**结论**：前端存储更适合无状态服务架构

---

### 3.3 智能上下文管理器（待实现）

## 四、双模式 Embedding 优化

### 4.1 存储两份向量

**优化方案**：
- 每个文档片段存储两份向量：
  - `semanticEmbedding`：Minimax API 生成的语义向量
  - `simpleEmbedding`：本地哈希生成的快速向量

**磁盘空间影响**：
- 每个向量是 384 维浮点数组（约 3KB）
- 存储两份向量增加约 3KB/文档
- 168 个文档增加约 504KB（可接受）

---

### 4.2 用户视角简化

**用户只需要感知**：
- **💡 语义检索（慢）**：理解问题含义，效果好（消耗 Token）
- **⚡ 快速检索（快）**：关键词匹配，速度快（不消耗 Token）

**不需要关心**：
- 上传模式的概念
- 向量存储的技术细节
- 查询模式与上传模式的匹配问题

---

### 4.3 兼容旧数据

**处理策略**：
- 旧数据只有 `embedding` 字段
- 系统自动将其转换为 `semanticEmbedding`
- `simpleEmbedding` 为空数组（兼容处理）

## 五、排查模型幻觉的方法

### 5.1 判断标准

**通过日志分析**：

1. **检索到的文档相关性**（distance 值）：
   - distance < 0.9：高相关性 → 知识库检索正确
   - distance > 1.5：低相关性 → 知识库检索问题

2. **回答是否基于知识库**：
   - 对比 `retrievedDocs` 和 `response`
   - 检查回答是否引用了检索到的内容

3. **多轮对话上下文**：
   - 查看 `messages` 字段
   - 了解历史对话是否影响当前回答

---

### 5.2 排查流程

```
1. 查看日志 → 分析 retrievedDocs 的 distance 值
2. 如果 distance 高 → 检索正确 → 可能是模型幻觉
3. 如果 distance 低 → 检索问题 → 优化知识库或检索参数
4. 查看 messages → 了解历史对话是否干扰
5. 查看 tokenUsage → 分析成本是否合理
```

## 七、总结

### 7.1 核心收获

1. **日志系统是必要的**：排查幻觉、分析成本、回放历史
2. **Token 消耗需要优化**：多轮对话会导致线性增长
3. **上下文管理是关键**：Minimax API 无状态，后端需要手动管理
4. **双模式 Embedding 是权衡**：用户简单，技术复杂

### 7.2 待优化项目

1. **智能上下文管理器**（高优先级）
   - 控制总上下文不超过 4000 tokens
   - 语义检索历史 + 滑动窗口 + 摘要压缩

2. **前端日志查看页面**（中优先级）
   - 日志列表、详情、统计图表

3. **检索精度优化**（中优先级）
   - 调整相似性阈值、优化返回数量
